# Закрытие сессии (end-session)

Это «ритуал закрытия» персональной базы знаний. Запускается, когда пользователь
заканчивает работу. Все действия подчиняются правилам из `AGENTS.md` в корне —
читай его, если есть сомнения по формату страниц, именованию или связям.

Выполняй шаги по порядку.

## Шаг 0. Сориентироваться

1. Прочитай `AGENTS.md` — это конституция (формат wiki-страниц, именование
   kebab-case, двусторонние связи, запрет на удаление из raw).
2. Прочитай `.second-brain-state.md` в корне — там лежит `last_commit`
   (SHA последнего задокументированного коммита) и `last_session` (дата).
   Если файла нет — считай это первой сессией.

## Шаг 1. Обработать raw/

1. Найди в `raw/` файлы **без** пометки `<!-- processed: YYYY-MM-DD -->`
   в начале.
2. Для каждого необработанного файла действуй по разделу «Обработка папки raw/»
   из `AGENTS.md`: читай целиком, выделяй отдельные сущности и идеи, создавай
   или обновляй страницы в `wiki/`, ставь взаимные ссылки.
3. После обработки добавь в начало исходного файла строку
   `<!-- processed: <сегодняшняя дата> -->`. **Файл не удаляй и не перемещай.**
4. Если необработанных файлов нет — пропусти шаг, отметь это в итоге.

## Шаг 2. Подтянуть изменения с GitHub

Документируем новые коммиты **этого** репозитория (red-wave) как знание о
развитии проекта.

1. `git fetch` — подтянуть свежие коммиты с remote (если нет сети — работай по
   локальному `git log`, отметь это в итоге).
2. Определи диапазон новых коммитов:
   - если в `.second-brain-state.md` есть `last_commit` —
     `git log <last_commit>..origin/main` (или `..HEAD`, если origin недоступен);
   - если состояния нет (первая сессия) — возьми разумный недавний диапазон,
     например `git log -n 20`, и отметь, что это первичная фиксация.
3. Для каждого значимого коммита (или группы связанных) реши, что это за знание:
   новая фича, изменение механики, фикс, рефактор, решение по архитектуре.
   Документируй **суть и причину**, а не дифф построчно.
   - Заводи/обновляй wiki-страницы по затронутым темам и сущностям
     (например, фича UI → страница про соответствующую систему; не плоди
     страницу на каждый коммит).
   - Ставь взаимные связи в секции «Связано с».
   - В секции «Источник» указывай коммит: короткий SHA и заголовок.
   - Мелкие/косметические коммиты можно агрегировать одной строкой, а не
     заводить под них отдельные страницы.

## Шаг 3. Противоречия

Если новое знание (из raw или из коммитов) противоречит тому, что уже записано
в wiki — **не перезаписывай молча.** По правилу из `AGENTS.md` отметь обе версии
и их источники прямо на странице, чтобы пользователь видел расхождение.

## Шаг 4. Обновить index.md

Обнови `index.md` под все изменения этой сессии: добавь новые страницы в разделы
«Темы» / «Сущности» / «Источники», поправь описания. Оглавление должно точно
соответствовать содержимому `wiki/`.

## Шаг 5. Записать CHANGELOG.md

Добавь сверху (под шапкой) запись за сегодня в формате:

```markdown
## YYYY-MM-DD
- что обработано из raw
- какие коммиты задокументированы (SHA + суть)
- какие страницы wiki созданы или обновлены
- какие новые связи установлены
```

Если запись за сегодня уже есть — дополни её, не дублируй заголовок даты.

## Шаг 6. Сохранить состояние

Перезапиши `.second-brain-state.md` в корне актуальными значениями:

```markdown
<!-- Состояние базы знаний. Файл служебный, правит его только end-session. -->
last_commit: <SHA HEAD/origin на момент закрытия>
last_session: <сегодняшняя дата>
```

## Шаг 7. Короткий итог пользователю

Выведи 4-6 строк:
- сколько файлов из raw обработано;
- сколько коммитов задокументировано;
- какие страницы wiki созданы/обновлены;
- какие новые связи установлены;
- замечены ли противоречия (если да — где).

## Важно про поиск знаний в будущем

Когда пользователь позже задаёт вопрос — **ищи ответ через базу знаний, а не
сканируй папки подряд**: сначала `index.md`, затем читай только релевантные
страницы `wiki/` (раздел «Ответы на вопросы» в `AGENTS.md`). Этот skill —
только про запись; чтение идёт по тем же правилам конституции.
